### Exact `__manifest__.py`

```python
{
    'name': 'Website Newsletter Block',
    'version': '18.0.1.0.0',
    'category': 'Website/Website',
    'summary': 'Newsletter subscription block for the website builder',
    'depends': ['website_mass_mailing'],
    'data': [
        'security/ir.model.access.csv',
        'views/snippets/snippets.xml',
    ],
    'assets': {},
    'installable': True,
    'auto_install': False,
    'license': 'LGPL-3',
}
```

**Why `assets: {}` not omitted:** An empty dict is valid and prevents a KeyError if any tooling reads the key. In Phase 2, asset bundles are added here. Omitting the key entirely is also valid.

**Why `category: 'Website/Website'`:** Matches `website_mass_mailing` (verified in upstream manifest). Determines where the module appears in the Apps list.

**Why `version: '18.0.1.0.0'`:** Odoo convention for community modules is `<odoo_major>.<module_version>` 5-part format. `odoo_brain` uses this format. Using `'1.0'` is also accepted but `18.0.1.0.0` is the correct convention for this dev environment.

## Architecture Patterns

### Recommended Project Structure (Phase 1)

```
odoo_website_newsletter_block/
├── __init__.py          # empty
├── __manifest__.py      # depends: [website_mass_mailing], data: [csv, xml]
├── security/
│   └── ir.model.access.csv
└── views/
    └── snippets/
        └── snippets.xml     # minimal <odoo/> placeholder
```

Everything else (`static/`, `controllers/`, `models/`) is deferred to Phase 2+.

### Pattern: Manifest `data:` Load Order

Odoo loads `data:` entries in list order. Security CSV must come before any XML that references model records.

```python
'data': [
    'security/ir.model.access.csv',   # load access rules first
    'views/snippets/snippets.xml',    # then view/template data
],
```

### Pattern: XMLID Naming for Access Records

The `id` column in the CSV becomes an XMLID in the module's namespace. Convention:

```
access_{model_name}_{descriptive_suffix}
```

Example: `access_mailing_list_newsletter_block`

Must be unique within the Odoo instance. The upstream module already uses `access_mailing_list_website_designer` — do not reuse that string.

## Common Pitfalls

### Pitfall 1: Missing `views/` placeholder causes install error

**What goes wrong:** Manifest lists `views/snippets/snippets.xml` in `data:` but the file doesn't exist → Odoo raises `FileNotFoundError` during module update, install fails with an ERROR log line (violates MODL-01).

**Prevention:** Always create every file referenced in `data:` and `assets:` before running `-u`. Even an empty `<odoo/>` XML wrapper is valid.

### Pitfall 2: Adding `google_recaptcha` or `website` to `depends` directly

**What goes wrong:** Not an install error, but violates MODL-02 (verified by the success criteria: "only `website_mass_mailing` listed"). These are already transitive dependencies — explicit listing is redundant and masks the intended minimal surface.

**Prevention:** `depends: ['website_mass_mailing']` only. Verify by grepping the manifest after writing it.

### Pitfall 3: XMLID collision on the access record `id`

**What goes wrong:** Using `access_mailing_list_website_designer` (the upstream module's id) causes a duplicate XMLID. On install Odoo may silently overwrite the upstream record, or in some versions raise a warning/error.

**Prevention:** Use a distinct `id`: `access_mailing_list_newsletter_block` (or similar). Distinct, unique within the Odoo instance.

### Pitfall 4: Non-empty `__init__.py` referencing missing directories

**What goes wrong:** If `__init__.py` contains `from . import controllers` but `controllers/` directory and its `__init__.py` don't exist, Python import fails with `ModuleNotFoundError` at Odoo startup. Module load fails with ERROR.

**Prevention:** Keep root `__init__.py` empty (or a comment) in Phase 1. Only add imports when the directories actually exist.

### Pitfall 5: Odoo binary path

**What goes wrong:** The success criteria run command is `odoo/bin/odoo -u odoo_website_newsletter_block`. The actual binary in this dev environment is `/Users/dortsman/Code/odoo18/odoo-bin`. Using the wrong path gives "command not found", which looks like a failure.

**The correct command for this environment:**
```bash
/Users/dortsman/Code/odoo18/odoo-bin -u odoo_website_newsletter_block --stop-after-init 2>&1 | grep -E "^[0-9].*ERROR|WARNING"
```
A clean install produces zero lines of output from that grep.

## Open Questions

1. **`version` string format**
   - What we know: `odoo_brain` uses `18.0.1.0.0`; `website_mass_mailing` uses `1.0`
   - What's unclear: Project has no stated preference
   - Recommendation: Use `18.0.1.0.0` to match the dev env's own module convention

2. **`auto_install` value**
   - What we know: `website_mass_mailing` uses `auto_install: ['website', 'mass_mailing']` (auto-installs when those are present); `odoo_brain` uses `False`
   - Recommendation: `False` — this is a custom client module, not a glue module. Should not auto-install.

3. **Whether `views/snippets/snippets.xml` placeholder can be omitted from Phase 1 manifest**
   - What we know: Manifest `data:` files must exist at install time
   - Recommendation: Include the placeholder XML from the start to avoid having to re-update the manifest mid-phase
