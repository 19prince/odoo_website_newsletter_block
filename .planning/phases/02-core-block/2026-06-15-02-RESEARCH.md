## Summary

Phase 2 delivers the complete QWeb template with all required CSS classes, data attributes, editable zones, and inline form — plus the XML registrations that make it draggable in the builder sidebar. No custom Python controller and no custom JS widget are needed: the existing `website_mass_mailing` module provides both. The only deliverables are XML (the snippet template, the sidebar registration xpath, and the snippet options xpath) and the manifest update to reference any new asset files.

The core mechanic is CSS class naming. The existing `publicWidget.registry.subscribe` auto-binds to `.js_subscribe`. The existing editor option `mailing_list_subscribe` writes `data-list-id` to `.s_newsletter_list`. The existing `is_subscriber` / `subscribe` JSON-RPC endpoints handle all backend logic. Our template just needs the right class names and data attributes in the right places — and they are all documented in the upstream source.

The critical design tension is that `_getListId()` has a hard-coded `[data-snippet=s_newsletter_block]` selector. The upstream template does NOT put this attribute on its section element — it relies on the fallback `this.$el.data('list-id')`. The safest resolution is to put `data-list-id="0"` directly on the `.js_subscribe` div (which also carries `s_newsletter_list`), making the fallback succeed regardless of whether `data-snippet` is present.

**Primary recommendation:** Build a single `views/snippets/snippets.xml` file containing three template/xpath blocks: (1) the `s_newsletter_block` QWeb template body, (2) the `website.snippets` registration xpath, (3) the `website.snippet_options` options xpath. Update `__manifest__.py` to add the JS bundle entries. No custom JS, no custom controller.

### Pattern 2: Sidebar Registration XPath

**What:** Registers the template as a draggable snippet under "Contact & Forms".

```xml
<!-- Source: /Users/dortsman/Code/odoo18/addons/website_mass_mailing/views/snippets_templates.xml line 7-19 -->
<template id="snippets" inherit_id="website.snippets">
    <xpath expr="//t[@id='mass_mailing_newsletter_block_hook']" position="after">
        <t t-snippet="odoo_website_newsletter_block.s_newsletter_block"
           string="Newsletter Block"
           t-forbid-sanitize="form"
           group="contact_and_forms">
            <keywords>newsletter, subscribe, signup, email, form</keywords>
        </t>
    </xpath>
</template>
```

**Why `position="after"` not `position="replace"`:** The upstream `website_mass_mailing` module already consumed the `mass_mailing_newsletter_block_hook` with `position="replace"`. Our module must not replace it again — that would remove the upstream newsletter block. Use `position="after"` to insert alongside it.

**Why `group="contact_and_forms"`:** Confirmed in `website/views/snippets/snippets.xml` line 27 — this is the correct group ID for the "Contact & Forms" sidebar category. Confirmed at line 344: `mass_mailing_newsletter_block_hook` is inside the `contact_and_forms` group block.

### Pattern 3: Snippet Options XPath

**What:** Wires the mailing list picker in the snippet options panel (LIST-02). Reuses the existing `mailing_list_subscribe` editor JS class — no custom JS needed.

```xml
<!-- Source: /Users/dortsman/Code/odoo18/addons/website_mass_mailing/views/snippets_templates.xml line 198-241 -->
<template id="newsletter_block_options" name="Newsletter Block Options"
          inherit_id="website.snippet_options">
    <xpath expr="//div[1]" position="before">
        <div data-js="mailing_list_subscribe"
             data-selector=".s_newsletter_list"
             data-target=".s_newsletter_list">
            <we-select string="Newsletter"
                       data-attribute-name="listId"
                       data-dependencies="!form_opt">
            </we-select>
        </div>
    </xpath>
</template>
```

**How it wires:** `data-js="mailing_list_subscribe"` loads `options.registry.mailing_list_subscribe` (defined in `website_mass_mailing.editor.js`). `data-selector=".s_newsletter_list"` means the panel activates when an element with that class is selected. `data-attribute-name="listId"` tells the picker to write `dataset.listId` (= `data-list-id`) on the `data-target` element.

### Pattern 4: In-Place Editing

**What:** Odoo's website builder makes elements editable when they carry the right attributes.

**How it works in Odoo 18:**
- Blocks of text inside `oe_structure` divs are editable by default.
- Individual elements can be marked editable by placing them inside a container that has `data-oe-model` / `data-oe-field` (for saved field content) OR by simply being within an editable region.
- For snippet text zones, Odoo's builder makes rich-text elements (h1–h6, p) directly editable within the snippet body without any extra attribute, provided the snippet body is in an `oe_structure` or the snippet is in the page's editable region.
- The `data-name` attribute on a container helps the editor label the zone — not required for editability.

**Verified behavior from upstream template:**
The upstream `s_newsletter_block_default_template` (line 51-61) contains bare `<h2>` and `<p>` elements inside the snippet body. No `contenteditable` or `data-oe-expression` attributes are on them. The builder makes them editable by virtue of being in the editable page region.

**Conclusion:** For EDIT-01 through EDIT-04, the text elements (`<h2>`, body `<p>`, sub-caption `<p>`, thank-you `<p>`) become editable in the builder without any special attribute — the builder's standard behavior applies to all text in the snippet body. The thank-you message inside `.js_subscribed_wrap` will be editable when the editor toggles the preview mode (the `toggleThanksMessage` option in `mailing_list_subscribe` exposes it).

### Anti-Patterns to Avoid

- **Putting `data-list-id` only on the outer `<section>`:** The editor option writes to `.s_newsletter_list` and `_getListId()` falls back to `this.$el.data('list-id')` — if neither has the attribute, `list_id` is undefined on submit.
- **Using `position="replace"` on `mass_mailing_newsletter_block_hook`:** Already consumed by upstream; replacing it removes the upstream block.
- **Putting editor JS in `web.assets_frontend`:** `options.Class` is not available on the frontend; loads `@web_editor/js/editor/snippets.options` which is editor-only. (Not applicable for Phase 2 since we're reusing upstream editor JS with no new file.)
- **Omitting `t-forbid-sanitize="form"`:** Silently strips `<input>` and `<button>` on first save. Fields present in edit mode, gone after save-and-reload.
- **Adding a name `<input>` without `name="name"`:** The existing subscribe click handler sends `value: $input.val()` from `.js_subscribe_value`. The name field is a plain `name="name"` input — NOT `js_subscribe_value`. The existing widget only picks up the email via `.js_subscribe_value`. The name input is submitted as a regular form field. **Important:** the existing `publicWidget.registry.subscribe` does NOT automatically pass the name to the controller (it sends only `value` = email). This means `address_name` will be null. Phase 2 accepts this limitation — name wiring is Phase 3. Confirmed out-of-scope by REQUIREMENTS.md traceability: FORM-02 (name captured) is Phase 3.

## Common Pitfalls

### Pitfall 1: `_getListId()` Hard-coded Selector Mismatch

**What goes wrong:** `_getListId()` tries `this.$el.closest('[data-snippet=s_newsletter_block')`. If `data-snippet="s_newsletter_block"` is absent from the section AND `data-list-id` is not on the `.js_subscribe` element, both paths return undefined. The subscribe RPC fires with `list_id: undefined`.

**Root cause:** The upstream `s_newsletter_block` template section (line 44) does NOT carry an explicit `data-snippet` attribute. The builder does not auto-stamp it either (verified by searching all builder JS — no code writes `dataset.snippet` to dropped elements). The JS widget works because the fallback `this.$el.data('list-id')` succeeds — `data-list-id="0"` IS on the `.js_subscribe` div via the embedded `s_newsletter_subscribe_form` template.

**How to avoid:** Put `data-list-id="0"` directly on the element that carries `s_newsletter_list js_subscribe` classes. This makes the fallback in `_getListId()` succeed. Optionally also add `data-snippet="s_newsletter_block"` to the outer section to enable the primary path.

**Warning signs:** Browser console shows `list_id: undefined` in RPC params; server throws on `int(None)`.

### Pitfall 2: `t-forbid-sanitize="form"` Missing from Registration

**What goes wrong:** Without it, the website's HTML sanitizer strips `<input>`, `<button>`, and `<a role="button">` from the saved page HTML on first save. The form renders correctly in edit mode; after save-and-reload, fields are gone.

**How to avoid:** Add `t-forbid-sanitize="form"` to the `<t t-snippet="...">` line in the `website.snippets` xpath. Verify by: (a) dropping the snippet, (b) saving the page, (c) reloading — inputs must still be present. Source: `snippets_templates.xml` line 9.

**Warning signs:** Snippet renders correctly in editor, form inputs missing on frontend after save.

### Pitfall 3: `s_newsletter_list` Class Missing or on Wrong Element

**What goes wrong:** The editor option `data-selector=".s_newsletter_list"` won't find its target. The mailing list picker panel either doesn't activate or writes `data-list-id` to the wrong element.

**How to avoid:** The element carrying `s_newsletter_list` must be the same element that carries `js_subscribe` and `data-list-id`. That's the inner div, not the outer `<section>`.

**Warning signs:** Snippet options panel doesn't show a mailing list picker when the block is selected; or list picker appears to work but `data-list-id` doesn't update on save.

### Pitfall 4: `position="replace"` on the Hook (Already Consumed by Upstream)

**What goes wrong:** `website_mass_mailing` already replaced `mass_mailing_newsletter_block_hook` with its own snippet. A second `position="replace"` on the same XPath target removes the upstream block from the sidebar.

**How to avoid:** Use `position="after"` to insert alongside the upstream block. Confirm by checking the builder sidebar — both "Newsletter Block" entries should appear.

**Warning signs:** Upstream "Newsletter Block" disappears from builder sidebar; only custom block is present.

### Pitfall 5: `o_cc` Without Specific Level (`o_cc1`–`o_cc5`)

**What goes wrong:** `o_cc` alone marks the element as a color combination zone but doesn't set a visual palette. The section appears with default browser styles, not the theme palette.

**How to avoid:** Use `o_cc o_cc2` (the level used by the upstream newsletter block). The editor's ColorCombination option allows the user to change the level after drop.

**Warning signs:** Snippet colors don't change when the user selects a palette swatch; or the block looks unstyled on all themes.

### Pitfall 6: Name Input Placeholder for Phase 2 — Not Wired to Controller

**What goes wrong (understanding, not a build failure):** The name `<input>` will be present in the form but the existing `publicWidget.registry.subscribe` does NOT pass it to the controller. The controller will create `mailing.contact` with `name = null`. This is expected behavior for Phase 2 — FORM-02 (name captured) is Phase 3.

**How to avoid (Phase 2):** Include the name input in the template so the form visually collects a name. Accept that it won't be persisted until Phase 3. Do not try to wire it in Phase 2.

## State of the Art

| Old/Wrong Approach | Correct Odoo 18 Approach | Notes |
|-------------------|--------------------------|-------|
| `position="replace"` on existing hook | `position="after"` when hook is already consumed | Upstream already consumed `mass_mailing_newsletter_block_hook` |
| Custom `publicWidget` for subscribe | Reuse `.js_subscribe` selector class — widget auto-binds | COMPAT-01 forbids new publicWidget patching |

| `contenteditable="true"` on all text zones | No attribute needed — builder edits text in snippet body by default | Verified by upstream template having bare `<h2>` and `<p>` without any extra attrs |
| `t-snippet-call` to embed the form | Flat template with all elements inline | Simpler, no template nesting confusion |

**Not deprecated but phase-deferred:**
- `address_name` wiring — controller already accepts it; widget doesn't pass it yet — Phase 3
- `list_id=0` guard — no guard in Phase 2; Phase 3 adds safety for unconfig'd blocks

## Open Questions

1. **Does `position="after"` on `mass_mailing_newsletter_block_hook` work when hook was already replaced?**
   - What we know: `website_mass_mailing` used `position="replace"` to consume the hook. After replacement, the `<t id="mass_mailing_newsletter_block_hook"/>` element no longer exists in the DOM.
   - What's unclear: Whether Odoo XPath inheritance processes in load order and whether `position="after"` can target the replacement element.
   - Recommendation: Test on install. Alternative if `position="after"` fails: use a broader xpath like `//snippets[@id='snippet_structure']//t[@group='contact_and_forms'][last()]` with `position="after"`, or use `//t[@id='mass_mailing_newsletter_popup_hook']` with `position="before"`.

2. **Does the builder's `ColorCombination` option automatically apply to the section with `o_cc o_cc2`?**
   - What we know: The upstream `s_newsletter_block` uses these classes and the color picker responds to it in the editor without any explicit option configuration in the XML.
   - Recommendation: Test by dropping the block and clicking it in the editor — a color palette row should appear in the sidebar. If not, add explicit `<div data-js="ColorCombination" data-selector=".s_newsletter_block">` in the options xpath.
