Append a third `<template>` block inside the `<odoo>` root of views/snippets/snippets.xml, after the closing `</template>` of the existing `snippets` template and before the closing `</odoo>` tag.

**Why a new template (not modifying existing):** The upstream `website_mass_mailing` snippet options template excludes `.s_newsletter_block .s_newsletter_list` from its `mailing_list_subscribe` selector. Our inner div is `.js_subscribe` (not `.s_newsletter_list`), but we still need our own registration to avoid collision. A dedicated template is clean and isolated.

**Exact content to add:**

```xml
<!-- ============================================================
     3. Snippet options â€” mailing list picker for LIST-01
     Upstream newsletter_subscribe_options_common excludes
     .s_newsletter_block .s_newsletter_list, so we register
     mailing_list_subscribe directly for our .js_subscribe div.
     data-attribute-name="listId" is camelCase â€” the options engine
     translates it to data-list-id on write (case-sensitive, must match exactly).
     ============================================================ -->
<template id="newsletter_block_options" inherit_id="website.snippet_options">
    <xpath expr="//div[1]" position="before">
        <div data-js="mailing_list_subscribe"
             data-selector=".s_newsletter_block .js_subscribe">
            <we-select string="Newsletter" data-attribute-name="listId" data-dependencies="!form_opt"/>
        </div>
    </xpath>
</template>
```

Key points:
- `data-selector=".s_newsletter_block .js_subscribe"` â€” targets our inner div. The upstream exclude only applies to `.s_newsletter_list` descendants, not `.js_subscribe`.
- No `data-target` attribute â€” `$target` defaults to the matched element (`.js_subscribe`), which is where `_getListId()` reads `data-list-id` from. This is correct.
- `data-attribute-name="listId"` â€” camelCase exactly as `mailing_list_subscribe._renderCustomXML` queries it: `uiFragment.querySelector('we-select[data-attribute-name="listId"]')`. Wrong case = picker renders but selections don't write data-list-id.
- `data-dependencies="!form_opt"` â€” matches upstream pattern; hides the select when form_opt is active (not applicable for our block, but harmless to include).
- No reCaptcha `<div>` needed â€” upstream already registers recaptchaSubscribe for `.s_newsletter_list`; adding it for our selector risks double-rendering. Omit for now.
