**Key class mechanics verified in source:**

| Class / attribute | Role |
|---|---|
| `s_newsletter_list` | Required — the `publicWidget.registry.subscribe` in `website_mass_mailing.js` targets `.js_subscribe` but `fixNewsletterListClass` adds this class retroactively; include it from the start |
| `js_subscribe` | Activates the frontend subscribe widget |
| `js_subscribe_value` | The email `<input>` — widget reads `.val()` from this |
| `js_subscribed_wrap` | Shown after successful subscribe; starts `d-none` |
| `js_subscribe_wrap` | Shown before subscribe; toggled by widget |
| `js_subscribe_btn` | The button the widget binds `click` to |
| `o_submit` | Marks the button as a submit action |
| `data-list-id="0"` | Initial value — overwritten by editor when designer picks a list |
| `o_cc o_cc2` | Color palette on the `<section>` — fully editable by the color picker in the editor |

**How `data-list-id` flows:**
1. Template sets `data-list-id="0"` on the `<section>` as a placeholder
2. The editor option `mailing_list_subscribe` (from `website_mass_mailing.editor.js`) renders a `<we-select>` populated with public mailing lists via `mailing.list.name_search`
3. When designer selects a list, `data-attribute-name="listId"` writes the value as `data-list-id` on the element matched by `data-selector=".s_newsletter_block"`
4. Frontend `_getListId()` in `website_mass_mailing.js` reads it: `this.$el.closest('[data-snippet=s_newsletter_block').data('list-id')` — note the `data-snippet` attribute on the `<section>` is required for this selector to work

**Critical:** You do NOT need to write your own `mailing_list_subscribe` JS option class — it already exists in `website_mass_mailing`. You only need the QWeb `<div data-js="mailing_list_subscribe" data-selector=".s_newsletter_block">` entry in `snippet_options` to wire your snippet to the existing editor option.

## Controller Pattern

Since `website_mass_mailing` already provides `/website_mass_mailing/subscribe` (JSON, public, website=True), you have two paths:

**Option A (recommended): Reuse the existing controller entirely**
- Point your form to the existing `/website_mass_mailing/subscribe` endpoint
- Your `js_subscribe` + `js_subscribe_btn` classes trigger the existing frontend widget which calls that route
- No custom controller needed at all
- Your `controllers/` directory can be omitted

**Option B: Custom `/website/newsletter/subscribe` route**
Only needed if you need different business logic (e.g., subscribing to multiple lists, custom validation).

```python
# controllers/main.py
from odoo.http import route, request
from odoo import _

class NewsletterController:

    @route('/website/newsletter/subscribe', type='json', website=True, auth='public', methods=['POST'])
    def subscribe(self, list_id, email, **kwargs):
        Contacts = request.env['mailing.contact'].sudo()
        Subscriptions = request.env['mailing.subscription'].sudo()

        contact = Contacts.search([('email', '=', email)], limit=1)
        if not contact:
            contact = Contacts.create({'name': email, 'email': email})

        existing = Subscriptions.search([
            ('contact_id', '=', contact.id),
            ('list_id', '=', int(list_id))
        ], limit=1)
        if not existing:
            Subscriptions.create({'contact_id': contact.id, 'list_id': int(list_id)})
        elif existing.opt_out:
            existing.opt_out = False

        return {'success': True, 'message': _("Thanks for subscribing!")}
```

```python
# controllers/__init__.py
from . import main

# __init__.py (top-level)
from . import controllers
```

**Route decorator notes:**
- `type='json'` — Odoo parses JSON body; JS must use `rpc()` not plain `fetch`. If you want a POST form submission, use `type='http'` and `methods=['POST']` instead.
- `website=True` — enables multi-website awareness and sets `request.website`
- `auth='public'` — allows unauthenticated visitors
- `csrf=True` is the default; do not disable it

## v18/v19 Compatibility Notes

**Verified safe (no version-specific internals):**

| Item | v18 status | v19 risk |
|---|---|---|
| `publicWidget` from `@web/legacy/js/public/public_widget` | Used in v18 source | MEDIUM risk — Odoo has been migrating away from legacy widgets toward Owl components. v19 may deprecate this path. |
| `rpc` from `@web/core/network/rpc` | Current import in v18 | LOW risk — this is the non-legacy RPC, stable |
| `options.Class.extend()` from `@web_editor/js/editor/snippets.options` | Used in v18 for editor options | MEDIUM risk — same legacy pattern concern as publicWidget |
| `@odoo-module` at top of JS files | Required in v18 | Required in v19 — safe |
| `data-list-id` attribute name | v18 source uses it | Safe — this is a data attribute you control |
| `o_cc` / `o_cc2` color palette classes | v18 source confirms them | Safe — CSS class names |
| `t-snippet` / `t-forbid-sanitize="form"` | v18 source uses it | Safe |
| `website.snippet_options` inherit pattern | v18 and prior versions | Safe — stable extension point |
| `inherit_id="website.snippets"` with hook xpaths | v18 source confirms hooks exist | Check: if `mass_mailing_newsletter_block_hook` is removed in v19 and this module replaces it, use a different xpath target |

**Specific avoidances for v18/v19 dual compatibility:**

1. **Do not use `this.$el` (jQuery) in new JS** if you can avoid it. v18 still ships jQuery but Odoo's direction is away from it. The existing `website_mass_mailing.js` uses it — reusing that code (via class hooks) is safer than writing new jQuery-dependent JS.

2. **Do not import from `@web/legacy/`** in new JS you write. Reuse existing widgets by targeting their CSS classes rather than extending them.

3. **Do not use `Widget.extend()`** — if you must write custom JS, use `publicWidget.Widget.extend()` only as a last resort. For this module, you likely need zero custom JS because the existing `subscribe` widget is activated by class names.

4. **`t-snippet-call` vs `t-call`** — v18 uses `t-snippet-call` for embedding sub-snippets that should be independently editable. Use `t-call` for structural sub-templates that are not separately draggable.

5. **The `data-snippet` attribute on `<section>`** — v18's `_getListId()` uses `closest('[data-snippet=s_newsletter_block')` (no closing bracket in the selector — this is a quirk in the source). Always set `data-snippet="s_newsletter_block"` on your outer `<section>` element.

6. **`groups="base.group_user"` on sub-templates** — `website_mass_mailing` uses this on `s_newsletter_block_default_template`. This restricts server-side rendering to logged-in users (the template content is embedded client-side in the editor, not rendered for public). If your module has editor-only sub-templates, apply this group restriction.
