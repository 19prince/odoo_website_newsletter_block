**Primary recommendation:** Add a controller override (Python), a widget override (JS), and a snippet options template (XML). Each change is narrow and isolated — no upstream files are modified.

### Pattern 1: Python Controller Override

Subclass `MassMailController`, re-decorate the same route. Odoo's routing resolves the most-derived class first.

```python
# controllers/main.py
from odoo import _
from odoo.http import route, request
from odoo.addons.website_mass_mailing.controllers.main import MassMailController

class NewsletterBlockController(MassMailController):

    @route('/website_mass_mailing/subscribe', type='json', website=True, auth='public')
    def subscribe(self, list_id, value, subscription_type, **post):
        # Guard: list not configured
        if int(list_id) == 0:
            return {
                'toast_type': 'danger',
                'toast_content': _("Please configure a mailing list for this block."),
            }
        # Recaptcha check (must consume token before calling super logic)
        if not request.env['ir.http']._verify_request_recaptcha_token('website_mass_mailing_subscribe'):
            return {
                'toast_type': 'danger',
                'toast_content': _("Suspicious activity detected by Google reCaptcha."),
            }
        fname = self._get_fname(subscription_type)
        address_name = post.get('address_name') or None
        self.subscribe_to_newsletter(subscription_type, value, list_id, fname, address_name=address_name)
        return {
            'toast_type': 'success',
            'toast_content': _("Thanks for subscribing!"),
        }
```

Key points:
- `_verify_request_recaptcha_token` pops `recaptcha_token_response` from `request.params` as a side-effect — must be called once, not deferred to super
- `post.get('address_name')` reads from `**post` kwargs (the JS sends it as a top-level RPC param)
- Do NOT call `super().subscribe(...)` — it would run recaptcha check a second time

### Pattern 2: JS Widget Override — include pattern

`publicWidget.Widget.extend({...})` creates a new class. To modify an existing registry entry, use `.include({...})`. This patches the prototype in place — all existing bindings pick up the change.

```javascript
/** @odoo-module **/
import publicWidget from "@web/legacy/js/public/public_widget";

publicWidget.registry.subscribe.include({
    /**
     * Override to add address_name from the name input to the subscribe RPC.
     */
    _onSubscribeClick: async function () {
        // Read name input before calling super (super triggers the RPC)
        const nameInputEl = this.el.querySelector('input[name="name"]');
        const addressName = nameInputEl ? nameInputEl.value.trim() : '';

        // Temporarily store on instance so we can inject in the RPC call.
        // We patch by overriding the full method and duplicating minimal logic.
        const self = this;
        const inputName = this.$('input').attr('name');
        const $input = this.$(".js_subscribe_value:visible, .js_subscribe_email:visible");

        if (inputName === 'email' && $input.length && !$input.val().match(/.+@.+/)) {
            this.$el.addClass('o_has_error').find('.form-control').addClass('is-invalid');
            return false;
        }
        this.$el.removeClass('o_has_error').find('.form-control').removeClass('is-invalid');

        const { rpc } = await odoo.loader.modules.get('@web/core/network/rpc');
        const tokenObj = await this._recaptcha.getToken('website_mass_mailing_subscribe');
        if (tokenObj.error) {
            self.notification.add(tokenObj.error, { type: 'danger', title: 'Error', sticky: true });
            return false;
        }
        rpc('/website_mass_mailing/subscribe', {
            'list_id': this._getListId(),
            'value': $input.length ? $input.val() : false,
            'subscription_type': inputName,
            'address_name': addressName,           // <-- addition
            recaptcha_token_response: tokenObj.token,
            turnstile_captcha: this.el.parentElement.querySelector('input[name="turnstile_captcha"]')?.value,
        }).then(function (result) {
            if (result.toast_type === 'success') {
                self._updateSubscribeControlsStatus(true);
                const $popup = self.$el.closest('.o_newsletter_modal');
                if ($popup.length) { $popup.modal('hide'); }
            }
            self.notification.add(result.toast_content, {
                type: result.toast_type,
                title: result.toast_type === 'success' ? 'Success' : 'Error',
                sticky: true,
            });
        });
    },
});
```

WARNING — `rpc` import: The upstream file imports `rpc` at module top-level with `import { rpc } from "@web/core/network/rpc"`. Use the same static import in our file, not `odoo.loader.modules.get()`. The dynamic loader approach shown above is illustrative but the static ESM import is cleaner and more correct.

Correct top-level form:
```javascript
/** @odoo-module **/
import publicWidget from "@web/legacy/js/public/public_widget";
import { rpc } from "@web/core/network/rpc";
import { _t } from "@web/core/l10n/translation";
```

### Pattern 3: Snippet Options Registration for LIST-01

The upstream `newsletter_subscribe_options_common` template registers `mailing_list_subscribe` for two selectors:
- `.o_newsletter_popup .s_newsletter_list` (popup)
- `.s_newsletter_list` excluding `.s_newsletter_block .s_newsletter_list` and `.o_newsletter_popup .s_newsletter_list`

Our inner `div.js_subscribe` is a descendant of `.s_newsletter_block`, and we cannot add `s_newsletter_list` to it without hitting the upstream exclude rule. Therefore we need our own options `<div>` entry.

The cleanest approach: add a new `<template>` block in snippets.xml that appends to `website.snippet_options`. Target our specific inner div by pairing `data-selector` with `data-exclude` pointing away from upstream blocks.

```xml
<!-- In views/snippets/snippets.xml, new <template> block -->
<template id="newsletter_block_options" inherit_id="website.snippet_options">
    <xpath expr="//div[1]" position="before">
        <!-- Mailing list picker for our custom newsletter block inner div -->
        <div data-js="mailing_list_subscribe"
             data-selector=".s_newsletter_block > .container .js_subscribe"
             data-target=".js_subscribe">
            <we-select string="Newsletter" data-attribute-name="listId" data-dependencies="!form_opt"/>
        </div>
        <!-- reCaptcha option for same target -->
        <div data-js="recaptchaSubscribe"
             data-selector=".s_newsletter_block > .container .js_subscribe"
             data-target=".js_subscribe">
            <t t-set="recaptcha_public_key" t-value="request.env['ir.config_parameter'].sudo().get_param('recaptcha_public_key')"/>
            <we-checkbox t-if="recaptcha_public_key" string="Show reCaptcha Policy" data-toggle-recaptcha-legal="" data-no-preview="true"/>
        </div>
    </xpath>
</template>
```

CAUTION on `data-attribute-name="listId"`: upstream `mailing_list_subscribe._renderCustomXML` looks for `we-select[data-attribute-name="listId"]`. This attribute name is case-sensitive and must match exactly. The JS then writes `data-list-id` on `this.$target` (the `.js_subscribe` div in our case, which already has `data-list-id="0"`).

CAUTION on `data-target`: when `data-target` is set, the JS reads/writes attributes on that element rather than on `this.$el` (the options div itself). We want writes to go to `.js_subscribe`, which is also `this.$target` when the user selects the `.js_subscribe` div. If the selector already matches `.js_subscribe` directly, `data-target` may be redundant — test both ways. Safest: omit `data-target` so `$target` defaults to the matched element.

### Pattern 4: NewsletterLayout option conflict avoidance

Upstream registers `NewsletterLayout` on `data-selector=".s_newsletter_block"`. Our section has class `s_newsletter_block`, so selecting our section in the builder will show a "Template" switcher (Email / Form). Clicking "Form Subscription" will replace our custom content with the upstream `s_newsletter_block_form_template`.

Two mitigation options:
- **Option A (recommended):** Add a CSS class to our section like `s_newsletter_block_custom` alongside `s_newsletter_block`, then register an options override that hides or replaces the Template selector for `.s_newsletter_block_custom`. This is Odoo-idiomatic but requires more XML.
- **Option B (simplest):** Accept the upstream Template switcher. Users who click it lose customization, which is acceptable for v1. Document it.
- **Option C:** Rename our section class entirely (remove `s_newsletter_block`). But `_getListId()` in the JS does `closest('[data-snippet=s_newsletter_block')` — our section already carries `data-snippet="s_newsletter_block"` via the builder stamp, so the class name change is safe at runtime. However, removing `s_newsletter_block` also removes `fixNewsletterListClass` activation — that widget adds `s_newsletter_list` to `.s_newsletter_block` elements, which is irrelevant for us since we don't use `s_newsletter_list`.

**Recommendation: Option B for Phase 3.** The risk is low (editors would have to deliberately click Template > Form Subscription). Document in handoff notes. Option A can be deferred to v2.

### Anti-Patterns to Avoid

- **Calling `super().subscribe()` in the controller override:** The recaptcha check pops `recaptcha_token_response` from `request.params` as a side-effect. Calling super after already calling `_verify_request_recaptcha_token()` yourself would try to check a now-missing token and fail.
- **Adding `s_newsletter_list` to the inner `.js_subscribe` div:** It IS a descendant of `.s_newsletter_block`, so it hits the upstream exclude rule in `newsletter_subscribe_options_common`. The picker would NOT appear. Use our own options registration (Pattern 3) instead.
- **Using `options.registry.mailing_list_subscribe.include({})` to change its selector:** The options registry data-selector is set at template render time from XML, not in JS. Changing the JS class won't reroute it. Fix it in XML.
- **Passing `address_name` as a separate URL param or form field:** The RPC route is `type='json'` — all params arrive as a JSON body. Add `address_name` to the JSON dict in the RPC call; it arrives in `**post` on the server.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Mailing list picker UI | Custom `we-select` population code | Inherit `mailing_list_subscribe` options class via XML data-selector | The JS already calls `orm.call('mailing.list', 'name_search', ...)` and populates the select; just make the selector hit our element |
| Contact deduplication | Custom create-or-find logic | `subscribe_to_newsletter()` static method | Already handles search-then-create, opt_out reactivation, session persistence |
| reCaptcha verification | Custom token check | `request.env['ir.http']._verify_request_recaptcha_token(action)` | Handles all result codes (no_secret = pass, timeout, wrong_token, etc.) |
| Success/error toast UI | Custom notification | Return `{'toast_type': ..., 'toast_content': ...}` dict | Upstream `_onSubscribeClick` reads these keys and passes to `self.notification.add()` |
| Subscription type detection | Read `name` attr | Keep email input DOM-first (already done in Phase 2) | Widget reads `this.el.querySelector('input').name` — email-first is the fix |

## Common Pitfalls

### Pitfall 1: Recaptcha token consumed twice

**What goes wrong:** If you call `super().subscribe()` after calling `_verify_request_recaptcha_token()` yourself, the token has already been popped from `request.params`. The super call finds no token and returns a recaptcha error.

**Why it happens:** `_verify_request_recaptcha_token` pops `recaptcha_token_response` from `request.params` as a side-effect (line 39 of `ir_http.py`): `token = request.params.pop('recaptcha_token_response', False)`.

**How to avoid:** Do not call super. Duplicate only the logic you need (the guard + fname + subscribe call). The proposed controller (Pattern 1) does this correctly.

### Pitfall 2: `address_name` not arriving in `**post`

**What goes wrong:** Controller receives `None` for `address_name` even though JS sends it.

**Why it happens:** The route signature is `def subscribe(self, list_id, value, subscription_type, **post)`. Any param not listed explicitly lands in `**post`. If JS sends `address_name` at the top level of the JSON body dict, it arrives as `post['address_name']`. If accidentally nested, it won't be found.

**How to avoid:** Ensure the JS sends `'address_name': addressName` as a flat key in the RPC params dict (same level as `list_id`, `value`, etc.). Verify with a `request.env['ir.logging']` debug log in the controller during development.

### Pitfall 3: `mailing_list_subscribe` picker doesn't appear for our block

**What goes wrong:** Editor selects our block in the builder, opens CUSTOMIZE panel — no Newsletter list selector appears.

**Why it happens:** The upstream options template excludes `.s_newsletter_block .s_newsletter_list`. Our inner div cannot have `s_newsletter_list` without hitting that exclude.

**How to avoid:** Register our own snippet options `<div>` with a selector specific enough to target our `.js_subscribe` but not overlap upstream blocks. See Pattern 3.

**Warning signs:** Inspect the CUSTOMIZE panel when our block section is selected. If no "Newsletter" select appears, the options div isn't matching.

### Pitfall 4: `data-attribute-name="listId"` case sensitivity

**What goes wrong:** List picker dropdown renders but selecting a list doesn't write `data-list-id` — subscription always uses list 0.

**Why it happens:** `mailing_list_subscribe._renderCustomXML` does `uiFragment.querySelector('we-select[data-attribute-name="listId"]')` — camelCase `listId`, not `list-id`. If the attribute in our options XML uses a different name, the select element is not found, buttons are not appended, and no selection event fires.

**How to avoid:** Use exactly `data-attribute-name="listId"` in the `we-select` element. The Odoo options system translates `listId` → `data-list-id` on write automatically (camelCase to kebab-case).

### Pitfall 5: `int(list_id) == 0` type coercion

**What goes wrong:** Guard check passes even when list isn't configured because `list_id` arrives as the string `"0"` rather than integer `0`.

**Why it happens:** JSON RPC params from JS `_getListId()` returns `this.$el.data('list-id')` — jQuery's `.data()` auto-parses to integer if the string is numeric. The value arrives server-side as Python `int`. But defensive coding is cheap.

**How to avoid:** Always cast: `if int(list_id) == 0`. This works whether `list_id` comes in as `0` (int) or `"0"` (str).

### Pitfall 6: Assets not reloaded after `__manifest__.py` change

**What goes wrong:** New JS file exists but widget override never runs.

**Why it happens:** Odoo caches asset bundles. Adding a new JS file to `web.assets_frontend` requires clearing the asset cache.

**How to avoid:** After adding the JS to manifest, restart Odoo with `-u odoo_website_newsletter_block` and clear browser cache (or open in incognito).

## State of the Art

| Old Approach | Current Approach | Impact |
|--------------|------------------|--------|
| `publicWidget.registry.X = publicWidget.Widget.extend({...})` | `publicWidget.registry.X.include({...})` | include patches prototype in-place; extend creates a new class that doesn't replace the registered one |
| Odoo 16: `address_name` wiring required a separate controller | Odoo 18: `subscribe_to_newsletter()` already accepts `address_name` parameter | We only need to (a) forward the param from `subscribe()` and (b) send it from JS |
| Custom options class JS | Inherit `mailing_list_subscribe` via XML `data-selector` targeting | No JS needed for the picker UI — existing options class handles everything |

## Open Questions

1. **Snippet options selector specificity**
   - What we know: `data-selector=".s_newsletter_block > .container .js_subscribe"` should uniquely match our block
   - What's unclear: whether Odoo's options engine supports CSS descendant selectors with `>` and spaces, or only simple class selectors
   - Recommendation: test with the simplest selector first (`.js_subscribe` with `data-exclude` listing upstream block selectors). Fall back to exact descendant path if needed.

2. **`data-list-id` write target for `mailing_list_subscribe`**
   - What we know: `_renderCustomXML` builds `we-button` elements with `data-select-data-attribute=mailingListId`. On selection, Odoo options engine writes the value to `data-list-id` on `this.$target`.
   - What's unclear: whether `this.$target` is the element matched by `data-selector`, or the element matched by `data-target` if different.
   - Recommendation: Since `_getListId()` in the widget reads from `.js_subscribe` (the fallback path), write the attribute there. Our options `<div>` should target `.js_subscribe` directly with no separate `data-target`.

3. **NewsletterLayout Template switcher conflict**
   - What we know: Selecting our section in the builder will show "Template: Email / Form" toggle (upstream `NewsletterLayout` targets `.s_newsletter_block`).
   - What's unclear: How disruptive this is in practice — does clicking "Form" immediately wipe content or prompt first?
   - Recommendation: Defer to Phase 3 test. If destructive without prompt, mitigate in Phase 3 by adding `data-newsletter-template="email"` to the section so the default is pinned and the switcher renders inactive.
