## Critical Pitfalls

### 1. `_getListId` Has a Hard-coded Snippet Name — It Won't Find Your Selector

**Source:** `website_mass_mailing/static/src/js/website_mass_mailing.js` line 113

```js
_getListId: function () {
    return this.$el.closest('[data-snippet=s_newsletter_block').data('list-id') || this.$el.data('list-id');
},
```

The selector `[data-snippet=s_newsletter_block` (note: missing closing `]`, but browsers tolerate this) is hard-coded. When the website builder drops a snippet from the panel, it auto-stamps `data-snippet` with the template ID. Your custom snippet will have `data-snippet="s_newsletter_custom_block"` (or whatever you name it), not `s_newsletter_block`.

**Consequence:** `_getListId` tries `closest('[data-snippet=s_newsletter_block')` first, gets `null`, then falls back to `this.$el.data('list-id')` — which only works if `data-list-id` is on the `.js_subscribe` element itself, not on the outer section. If you mirror the standard structure (list-id on the outer section, `js_subscribe` widget nested inside), the fallback won't work either.

**Warning sign:** Form submits with `list_id: undefined` or `list_id: null`; server throws an uncaught exception trying `int(None)`.

**Prevention:**
- Put `data-list-id` directly on the `.js_subscribe` element (not just the outer section), OR
- Override `_getListId` in your own JS widget subclass to use the correct selector, OR
- Name your root section class to also include `s_newsletter_list` and put `data-list-id` there — the widget walks up via `closest`, so the first matching ancestor wins.

The safest approach: keep `data-list-id` on the `.js_subscribe` div directly, making the fallback in core's `_getListId` work without any override.

**Phase:** Phase 1 (Template + JS foundation)

### 2. `list_id=0` Passed to Subscribe Endpoint Causes a Postgres FK Violation

**Source:** `website_mass_mailing/controllers/main.py` lines 51–68

The `subscribe_to_newsletter` static method does `int(list_id)` and then passes the result directly to `ContactSubscription.create({'contact_id': ..., 'list_id': int(list_id)})`. There is **no guard** for `list_id == 0`.

`mailing.subscription` has `list_id = Many2one('mailing.list', required=True)`. Passing `list_id=0` to ORM create generates a Postgres FK constraint violation — an unhandled 500 error that bubbles up as an opaque JSON-RPC error to the client.

The editor JS `onBuilt` handler does set `data-list-id` to the first mailing list (line 20), and removes the snippet if no list exists (line 22–39). But this only fires in the website builder. A snippet dropped and then saved without a list configured, or a snippet whose `data-list-id` was not updated correctly, will reach the controller with `list_id="0"`.

**Warning sign:** Server logs show `psycopg2.errors.ForeignKeyViolation` on subscribe attempts; client gets a `danger` toast only if you add error handling, otherwise silent failure.

**Prevention:**
- Add an explicit guard at the top of your controller: `if not list_id or int(list_id) == 0: return {'toast_type': 'danger', 'toast_content': _("No mailing list configured.")}`.
- Also guard the `is_subscriber` check the same way (currently it would do `search_count` with `list_id in [0]`, which returns 0 cleanly, so less dangerous but semantically wrong).

**Phase:** Phase 1 (Controller)

### 3. The Editor Option Writes `data-list-id` to `.s_newsletter_list`, Not the Outer Section

**Source:** `views/snippets_templates.xml` lines 213–230

The `mailing_list_subscribe` option block has `data-target=".s_newsletter_list"`. When the editor writes the selected mailing list ID, it writes `data-list-id` on the element matching `.s_newsletter_list` — which in the standard template is the **inner form div** (class `s_newsletter_subscribe_form s_newsletter_list js_subscribe`), not the outer `<section>`.

If your custom template puts `data-list-id` only on the outer `<section>` and your `.js_subscribe` element doesn't have the `s_newsletter_list` class, the editor option will fail to find its target and silently not update the attribute.

**Warning sign:** Selecting a mailing list in the editor panel appears to work but the `data-list-id` doesn't change on the saved page.

**Prevention:**
- Add `s_newsletter_list` class to your `.js_subscribe` wrapper div (the one with `data-list-id`).
- Keep `data-list-id="0"` on that same element as the default.
- Mirror the standard structure: the form/widget container is the `s_newsletter_list` element; the outer section is just for layout.

**Phase:** Phase 1 (Template structure)

### 4. Editor JS Goes in `website.assets_wysiwyg`, Frontend JS in `web.assets_frontend` — Mixing Them Breaks Both

**Source:** `website_mass_mailing/__manifest__.py` lines 21–36

The asset bundle split is:
- `web.assets_frontend` — loaded on the public site (the `publicWidget` for form submission)
- `website.assets_wysiwyg` — loaded only in the website builder/editor (snippet option classes)

If your `options.Class` (editor panel JS) ends up in `web.assets_frontend`, it loads on every page view and throws errors because `@web_editor/js/editor/snippets.options` is not available on the frontend. The reverse (putting `publicWidget` code in `assets_wysiwyg` only) means the form won't work for visitors.

**Warning sign:** JS console errors on the frontend like `Cannot read properties of undefined (reading 'Class')`, or snippet options panel that never populates in the editor.

**Prevention:**
- `static/src/js/website_newsletter_block.js` (publicWidget, form submission) → `web.assets_frontend`
- `static/src/js/website_newsletter_block.editor.js` (options.Class) → `website.assets_wysiwyg`
- Any XML templates needed by the editor → `web.assets_frontend` (they're needed by the frontend widget's XML too) — check `website_mass_mailing/__manifest__.py` line 26: `'website_mass_mailing/static/src/xml/*.xml'` is in `assets_frontend`.

**Phase:** Phase 1 (Module scaffolding)

### 5. `data-js` Value Must Exactly Match the `options.registry` Key (Case-Sensitive)

**Source:** `views/snippets_templates.xml` line 227 + `website_mass_mailing.editor.js` line 8

```xml
<div data-js="mailing_list_subscribe" ...>
```
```js
options.registry.mailing_list_subscribe = options.Class.extend({...
```

These must match exactly. The `data-js` attribute in the options XML tells the builder which `options.registry` entry to instantiate. A mismatch (e.g., `data-js="NewsletterSubscribe"` but `options.registry.newsletter_subscribe`) silently skips the option panel — no error, the panel just doesn't show your custom option.

**Warning sign:** No custom panel appears when the snippet is selected in the editor; no JS errors.

**Prevention:** Pick a name, use it identically in both places. Convention in Odoo 18 is camelCase for compound words (`NewsletterLayout`, `recaptchaSubscribe`) or snake_case (`mailing_list_subscribe`) — either works as long as both sides match.

**Phase:** Phase 1 (Editor option wiring)

## v18/v19 Compatibility Traps

### `publicWidget` Is Legacy Infrastructure, Not Deprecated — But Watch Import Paths

**Source:** `website_mass_mailing/static/src/js/website_mass_mailing.js` line 4

```js
import publicWidget from "@web/legacy/js/public/public_widget";
```

The path includes `/legacy/`. In v18 this is the correct, supported path for public widgets. However, Odoo's trajectory is toward Owl components everywhere. In v19, `publicWidget` may be further deprecated or the import path may change.

**Risk for v18/v19 compat:** LOW currently — the existing `website_mass_mailing` module uses it the same way in v18. If v19 drops it, Odoo will ship a compatibility shim or update their own modules first.

**Prevention:** Match the import path exactly from `website_mass_mailing.js`. Don't try to "modernize" by using Owl components for the public-facing widget — that would break v18.

---

### `bindService("notification")` Works in `publicWidget` via `ServicesMixin`

**Source:** `web/static/src/legacy/js/public/public_widget.js` line 336

`ServicesMixin.bindService` uses `Component.env.services` — it accesses the Owl service registry. This works in v18 because `publicWidget` roots are attached to an Owl component environment. It should continue working in v19 as long as `publicWidget` exists.

If you need to show toast notifications from your widget, use `this.notification = this.bindService("notification")` in `init()` — same pattern as core.

---

### `rpc` Import Path Changed in v16+ and Is Now Stable in v18

**Source:** `website_mass_mailing.js` line 7

```js
import { rpc } from "@web/core/network/rpc";
```

The old v15 path `web.rpc` / `@web/core/legacy/rpc` is gone. The current path is stable in v18 and expected to remain in v19. Don't use the old path.

### `trigger_up` Is a Legacy Event Bus — Fine for v18 Editor Options

**Source:** `website_mass_mailing.editor.js` lines 25, 34

```js
this.trigger_up("request_save", {...});
this.trigger_up("remove_snippet", {...});
```

`trigger_up` is the legacy widget event system. It works in v18 editor options because `options.Class` inherits from the legacy widget base. This is still the pattern used by Odoo's own modules.

### `is_public` on `mailing.list` Defaults to `False`

**Source:** `mass_mailing/models/mailing_list.py` line 40–45

```python
is_public = fields.Boolean(
    string='Show In Preferences', default=False, ...
)
```

The `_renderCustomXML` option fetches only lists where `is_public = True`. If a user creates a mailing list without checking "Show In Preferences", it won't appear in the snippet's dropdown — the dropdown will be empty, and `onBuilt` will offer to redirect the user to create one.

**Warning sign:** Editor shows "No mailing list found" dialog even though lists exist in the backend.

**Prevention:** Document in the module README that the target mailing list must have "Show In Preferences" enabled. Consider adding a hint in the `onBuilt` dialog text if you override it.

## Common Snippet Module Mistakes

### Missing `t-forbid-sanitize="form"` on Snippet Registration

**Source:** `website_mass_mailing/views/snippets_templates.xml` line 9

```xml
<t t-snippet="website_mass_mailing.s_newsletter_block" string="Newsletter Block" t-forbid-sanitize="form" group="contact_and_forms">
```

Without `t-forbid-sanitize="form"`, the website builder's HTML sanitizer strips form elements (`<input>`, `<button>`) when saving the snippet. Your form fields will disappear silently after the first save.

**Warning sign:** Fields render fine in edit mode but vanish after save-and-reload.

**Prevention:** Add `t-forbid-sanitize="form"` to your `t-snippet` declaration in the `website.snippets` template extension.

---

### Inheriting `website.snippets` at the Wrong XPath Hook

**Source:** `website_mass_mailing/views/snippets_templates.xml` lines 7–19

The snippets panel has named hook points (`t-id` attributes in `website.snippets`). If you try to insert at a nonexistent hook or use `position="inside"` on the wrong element, your snippet silently doesn't appear in the builder panel.

**Warning sign:** Module installs without error, but snippet isn't in the builder's "Add Block" panel.

**Prevention:** Inspect `website/views/snippets/snippets.xml` for the correct hook. Use `group="contact_and_forms"` (the category your snippet belongs to). The `mass_mailing_newsletter_block_hook` hook is already consumed by `website_mass_mailing`; your custom module should use a fresh `t-snippet` with its own string label, not replace the existing hook.

### Saving Preview Classes (`o_disable_preview`, `o_enable_preview`) to the Database

**Source:** `website_mass_mailing.editor.js` lines 44–51

```js
cleanForSave() {
    const previewClasses = ['o_disable_preview', 'o_enable_preview'];
    const toCleanEls = this.$target[0].querySelectorAll(".js_subscribe_wrap, .js_subscribed_wrap");
    toCleanEls.forEach(element => {
        element.classList.remove(...previewClasses);
    });
},
```

The editor uses `o_disable_preview` / `o_enable_preview` on `.js_subscribe_wrap` and `.js_subscribed_wrap` to preview the "thanks" state toggle while editing. Without `cleanForSave()` these classes persist in the saved HTML, causing the form to render in the wrong state for all visitors.

**Warning sign:** After saving with the "Display Thanks Message" toggle active, all visitors see the thanks state instead of the subscribe form.

**Prevention:** Implement `cleanForSave()` in your `options.Class` to strip any editor-only preview or state classes before save.

### Using `o_cc` Without Understanding Color Combination Inheritance

**Source:** `website_mass_mailing/views/snippets_templates.xml` line 44

```xml
<section class="s_newsletter_block s_newsletter_list o_cc o_cc2 pt64 pb64">
```

`o_cc` marks an element as a "color combination" container. `o_cc2` sets it to preset #2 (typically the secondary palette). Without `o_cc`, a section doesn't participate in the theme's color system — background, text, and button colors won't update when the user changes their theme palette.

Adding `o_cc` alone without a number (`o_cc1`–`o_cc5`) has no visual effect. The number is what selects the palette. `o_colored_level` (used on nested sections like `s_text_block`) inherits from the closest ancestor's `o_cc` level.

**Warning sign:** Snippet colors don't respond to the website's palette editor; or snippet looks wrong on all themes except the default.

**Prevention:** Follow the same pattern as `s_newsletter_block`: `o_cc o_cc2` on the outer section, and let nested elements inherit. Allow the editor option `ColorCombination` to change the number (1–5) if you want the user to control it.

## Controller / Form Submission Pitfalls

### Custom `type='http'` Controller Requires Manual CSRF Handling; `type='json'` Does Not

**Source:** `website/controllers/form.py` lines 31–38

```python
@http.route('/website/form/<string:model_name>', type='http', auth="public", methods=['POST'], website=True, csrf=False)
def website_form(self, model_name, ...):
    csrf_token = request.params.pop('csrf_token', None)
    if request.session.uid and not request.validate_csrf(csrf_token):
        ...
```

The native website form controller is `type='http'` with `csrf=False` and manually validates the token. This is because HTML forms POST to it and don't go through the JSON-RPC path.

If you build a **custom** controller at `/website/newsletter/subscribe`:
- `type='json'` → CSRF is automatically enforced via the JSON-RPC session token (no `csrf_token` field needed in the payload). Use `rpc(...)` from `@web/core/network/rpc` in JS.
- `type='http'` → Either set `csrf=False` and manually validate, or include a CSRF token from `request.csrf_token()` rendered in the template. If you forget and leave `csrf=True` on a public-facing POST, every submission will return 403.

The existing subscribe endpoint (`/website_mass_mailing/subscribe`) uses `type='json'`, which is the right pattern for JS-driven form submission. Match it.

**Prevention:** Use `type='json', auth='public', website=True` on your controller. Send via `rpc(...)` in JS. No CSRF token field needed.

---

### `type='json'` Controllers Return Python Dicts — Not HTTP Status Codes

**Source:** `website_mass_mailing/controllers/main.py` lines 38–48

```python
return {
    'toast_type': 'danger',
    'toast_content': _("Suspicious activity detected by Google reCaptcha."),
}
```

A `type='json'` controller that returns a dict is **always HTTP 200**. Error states are communicated in the payload (`toast_type: 'danger'`), not via HTTP status. If you raise an exception inside a `type='json'` controller, Odoo serializes it as a JSON-RPC error object, and the `rpc()` call will throw in JS — you need a `.catch()` or `try/catch`.

**Warning sign:** Form submission silently fails; no error shown because the JS doesn't handle the rejection.

**Prevention:** Return a success/error dict like the pattern above. Wrap `rpc(...)` in a `.catch(err => { ... })` handler in JS.

### Recaptcha Is Bypassed When No Key Is Configured — Don't Assume It's Always Active

**Source:** `google_recaptcha/models/ir_http.py` lines 72–74

```python
private_key = request.env['ir.config_parameter'].sudo().get_param('recaptcha_private_key')
if not private_key:
    return 'no_secret'
```

And in `_verify_request_recaptcha_token`: `if recaptcha_result in ['is_human', 'no_secret']: return True`.

No recaptcha private key configured → verification always returns `True` → subscribe endpoint allows all requests. This means in development (no key set) your controller works fine, but in production (with key set) you must pass the actual recaptcha token. The JS widget initializes `ReCaptcha` and calls `this._recaptcha.getToken(...)` in `_onSubscribeClick`. If recaptcha libs haven't loaded, `getToken` returns `{error: ...}` and the click handler returns early.

**Warning sign:** Works in dev, fails silently in prod.

**Prevention:** Test with a recaptcha key configured in your dev environment. Include `google_recaptcha` in your module's `depends` list (already there in `website_mass_mailing`, so inherited if you depend on it).

## Mailing List / Contact Pitfalls

### `mailing.contact` Does NOT Have a Unique Constraint on Email — Duplicates Are Silent

**Source:** `mass_mailing/models/mailing_contact.py` (no `_sql_constraints` on email), `website_mass_mailing/controllers/main.py` lines 63–67

```python
contact_id = Contacts.search([(fname, '=', value)], limit=1)
if not contact_id:
    contact_id = Contacts.create({'name': name, fname: value})
ContactSubscription.create({'contact_id': contact_id.id, 'list_id': int(list_id)})
```

The controller searches for an existing contact by email first (`limit=1`) — so it won't create a duplicate `mailing.contact` for the same email. However, `mailing.contact` has no DB-level unique constraint on `email`. If two requests race (concurrent submissions), two contacts with the same email can be created.

`mailing.subscription` has `unique (contact_id, list_id)` at DB level. So a race that creates two contacts and then tries to subscribe both to the same list will succeed (different `contact_id` values), resulting in two subscription records for the same email address.

**Warning sign:** Mailing list contact count appears doubled; emails received twice by the same recipient.

**Prevention:** This is a core behavior you can't easily fix without patching. For a custom controller, wrap the contact lookup + create in a `with self.env.cr.savepoint():` block, or use `try/except psycopg2.IntegrityError` if you add a unique constraint via your module.

For most use cases (low-volume newsletter signups) the race is rare enough to ignore. Document it.

### Re-subscribing a Previously Opted-Out Contact Silently Sets `opt_out=False`

**Source:** `website_mass_mailing/controllers/main.py` lines 68–70

```python
elif subscription.opt_out:
    subscription.opt_out = False
```

No error, no explicit confirmation — opt-out is reversed. This is the intended behavior but can surprise you if you're testing: if a contact is opted out in your test list, submitting the form again re-subscribes them without any error or special handling. The success toast fires the same as for a new subscription.

**Prevention:** Behavior is correct for a newsletter widget. No action needed, but document it so future maintainers don't treat this as a bug.

### The `name` Field on `mailing.contact` Will Be the Email Address When No Name Is Provided

**Source:** `website_mass_mailing/controllers/main.py` lines 55–57 + `odoo/tools/mail.py` line 951–961

```python
name, value = tools.parse_contact_from_email(value)
if not name:
    name = address_name  # address_name is None by default
```

`parse_contact_from_email("user@example.com")` returns `('', 'user@example.com')` — an empty name. `address_name` defaults to `None`. So `Contacts.create({'name': None, 'email': 'user@example.com'})` is called.

`mailing.contact.name` has no `required=True` — it's just a `Char` field. The record is created with an empty/null name. In the backend, these contacts show up with no name, which looks messy.

Your form collects a `name` field. Pass it as `address_name` to `subscribe_to_newsletter` (the static method's optional last parameter) so the contact gets named properly.

**Warning sign:** All contacts created by your snippet show empty names in the Email Marketing contact list.

**Prevention:** In your controller, extract the name from the POST payload and pass it as `address_name` parameter. The core `subscribe_to_newsletter` already handles this case.
