## What Existing `s_newsletter_block` Already Handles (Reuse for Free)

Do not reinvent these. Hook into them.

**Subscription backend (controller + model)**
- `/website_mass_mailing/subscribe` JSON-RPC endpoint — creates `mailing.contact`, creates `mailing.subscription`, handles opt-out reinstatement (`subscription.opt_out = False`). Source: `controllers/main.py` lines 51–72.
- `/website_mass_mailing/is_subscriber` — checks on page load whether the current session email is already subscribed; pre-fills the input and disables the button. Source: `controllers/main.py` lines 11–21.
- Duplicate email handling — if contact with same email exists, reuses it; if subscription exists, reinstates rather than duplicates. Source: `controllers/main.py` lines 61–70.
- Session persistence — after subscribe, stores email in `request.session['mass_mailing_email']` so the user appears subscribed on page reload. Source: line 72.

**JS widget (`publicWidget.registry.subscribe`)**
- Selector `.js_subscribe` — the widget auto-attaches to any element with this class.
- `_updateSubscribeControlsStatus(isSubscriber)` — toggles `.js_subscribe_wrap`/`.js_subscribed_wrap` visibility and disables input. This IS the inline swap mechanic. Source: `website_mass_mailing.js` lines 86–110.
- reCaptcha v3 integration — already wired; fires on submit, passes token to controller, controller calls `_verify_request_recaptcha_token`. Source: JS lines 132–139; controller line 37.
- Cloudflare Turnstile fallback — also wired in JS if `session.turnstile_site_key` is set. Source: JS lines 23–26 and 100–109.

**Editor infrastructure**
- `options.registry.mailing_list_subscribe` — renders a "Newsletter" dropdown in the editor sidebar populated with all `mailing.list` records where `is_public=True`. Auto-sets `data-list-id` on drop. Source: `website_mass_mailing.editor.js`.
- "Display Thanks Message" editor checkbox — toggles a preview of the thank-you state in edit mode without actually subscribing. Source: `editor.js` lines 60–68.
- `cleanForSave()` — strips preview-mode classes before saving to DB. Source: `editor.js` lines 44–51.
- "No mailing list" guard — if no public lists exist when snippet is dropped, prompts user to create one or removes the snippet. Source: `editor.js` lines 17–39.
- reCaptcha legal text toggle in editor sidebar. Source: `options.registry.recaptchaSubscribe`.

**Template switching (two modes already exist)**
- `email` mode: `s_newsletter_block_default_template` — headline + body text left, email-only subscribe form right. This is our starting point.
- `form` mode: `s_newsletter_block_form_template` — full `s_website_form` with name + email + list checkboxes + consent checkbox + end message. Uses `/website/form/` route instead.

## What We Must Build Fresh

Gaps in existing infrastructure that our block needs.

| Gap | Why It Doesn't Exist | What to Build |
|-----|---------------------|---------------|
| Name field in the simple subscribe widget | `s_newsletter_subscribe_form` (the `js_subscribe` path) only has an email input. Name is only available in the full `s_website_form` (`form` mode). | Add `<input name="name">` to our template; pass `address_name` to the controller subscribe call (already accepted as param: `controllers/main.py` line 51 has `address_name=None`). |
| Sub-caption slot | Neither template has one. | Add a `<p class="small text-muted">` below the form; mark `data-oe-expression` for editability. |
| Custom thank-you message (editable) | The existing `.js_subscribed_wrap` contains only a hardcoded "Thanks for registering!" `<p>`. | Replace with an editable content block inside `.js_subscribed_wrap` so site editors can customize the message. |
| Wiring `data-list-id` at the section level (not form level) | Existing `_getListId()` in the JS looks for `data-list-id` on the closest `[data-snippet=s_newsletter_block]` OR on the element itself. Our custom block must set it correctly so the widget can find it. | Set `data-list-id="0"` on the section; confirm the selector in `_getListId()` matches. |

## Differentiators (v2+)

Defer these. They add complexity without fixing anything broken in v1.

| Feature | Value | Why Defer |
|---------|-------|-----------|
| Multi-list selection (checkboxes) | Lets visitor pick which lists to join | The `form` mode already does this via `s_website_form`; adds significant template complexity to the simple widget path |
| GDPR consent checkbox | Explicit opt-in text + required checkbox | Adds a required field; increases friction; the existing `form` template already has it (`mailing_sub3`) — extract from there if needed |
| Double opt-in (confirmation email) | Cleaner list hygiene | Requires Odoo `mass_mailing.group_opt_in` config + email template; out of scope for block itself |
| Split first/last name | Already supported by `mailing.contact` (`_is_name_split_activated()`) | Config-dependent; adds two inputs |
| Turnstile-only mode without reCaptcha | Performance | Already works via session key; no block code needed |
| Success redirect to a URL instead of inline swap | Flexibility | Low demand for simple newsletter block use case |
| Phone/SMS subscription | `subscription_type='mobile'` already in the controller | Different UX pattern; separate block |

## Anti-Features

Explicitly do not build these.

| Anti-Feature | Why Not | What to Do Instead |
|--------------|---------|--------------------|
| Custom subscribe controller/route | The existing `/website_mass_mailing/subscribe` already handles create, dedup, opt-out reinstatement, reCaptcha, and session. Duplicating it means maintaining two code paths. | POST to the existing endpoint; pass `address_name` for the name field. |
| Custom `mailing.contact` write logic in the block module | `mailing.contact` minimum to create: only `name` and `email` (both optional at model level, but email is practically required for mass mailing to function). The controller already does the right thing. | Let the controller handle it. |
| Full `s_website_form` wrapper | That's the existing `form` mode of `s_newsletter_block`. Don't re-implement it. Use it as a reference but keep our block on the simpler `js_subscribe` path. | Keep the `.js_subscribe` widget path for simplicity. |
| Embedded mailing list management (unsubscribe UI) | That's `/mailing/my` — a full portal page. Out of scope for a signup block. | Link to `/mailing/my` in the sub-caption if needed. |
| Storing subscriber email in a custom model | `mailing.contact` + `mailing.subscription` is the canonical Odoo structure. A custom model creates islands. | Write to `mailing.contact` exclusively. |
| Client-side-only "subscribe" (no server call) | Worthless — no record created in Odoo. | Never. |
| Showing subscriber counts or social proof numbers in the block | Requires additional privileged DB reads on every page render; security risk. | Anti-feature. Omit. |

## GDPR / Compliance Notes

**What Odoo already provides (we get for free):**
- `mail.blacklist` model — unsubscribed emails are permanently blocked from all mailings. The controller does NOT check the blocklist at subscribe time (it will create the contact), but the mass-mailing send process excludes blacklisted addresses. Source: `mailing_list.py` line 247: `contact.email_normalized NOT IN (select email from mail_blacklist where active = TRUE)`.
- `/mailing/my` portal page — subscribers can manage their own list subscriptions and add themselves to the blocklist. Built-in. No action needed from us.
- `opt_out` flag on `mailing.subscription` — reinstated (not duplicated) by the controller when a returning subscriber re-submits.
- One-click unsubscribe route (`/mailing/<id>/unsubscribe_oneclick`) and confirmation flow. Built-in. Emails sent by Odoo include these links automatically.

**What we are responsible for:**
- Do NOT suppress or omit the reCaptcha / Turnstile wiring. The existing JS already fires this; our template must not break the `.js_subscribe` selector chain.
- The "I agree to receive updates" checkbox (present in `form` mode, absent in `email` mode) is NOT included in our simple block by default. If the target market is EU/GDPR-regulated, this is a v1 concern, not v2. Flag to Darren: the existing `s_newsletter_block_form_template` includes a required consent checkbox (`mailing_sub3`); our simpler block does not. Site owners in GDPR jurisdictions should either use the `form` mode or we add the checkbox as a v1 option.
- `is_public` flag on `mailing.list` — the editor's list selector (`mailing_list_subscribe` option) only shows `is_public=True` lists. The `website_form.py` controller actively rejects submissions to private lists (`private_list_ids` check). Ensure the editor option inherits this filter. Source: `website_form.py` lines 19–24.
- The `data-list-id="0"` initial state means submissions with a zero list-id hit the controller with `list_id=0`. The subscribe controller will try `int(list_id)` = 0, which finds no subscription and creates a contact with `list_id=0` subscription — a broken state. The editor guard (`onBuilt`) prevents this in normal use, but **the block must render visually disabled or show a placeholder when `data-list-id` is still 0** (e.g., before a list is configured). Source: editor.js lines 17–39 shows the existing `s_newsletter_block` has this guard but only in edit mode — publish mode with `list_id=0` is a silent data integrity issue. This is a must-fix for v1.
